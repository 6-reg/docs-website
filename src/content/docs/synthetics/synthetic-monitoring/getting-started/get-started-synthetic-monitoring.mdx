You can think of our synthetic monitors as crash test dummies for your websites, applications, and API endpoints. When you deploy a synthetic monitor, an API check or Chrome browser runs through your most important workflows, testing your critical endpoints and reporting the results to New Relic. By mimicking the behaviors of customers, your monitors test how well your web apps handle traffic. 

The below tutorial walks you through creating your first monitor. Each step shows the process to create one of our monitors, either a simple browser, step, or scripted browser monitor.

## Create a scripted browser monitor [#browser]

Scripted browser monitors let you test the availability, functionality, and performance of critical user flows in your web application. When your script runs, New Relic creates a fully virtualized Selenium-driven Google Chrome browser that navigates your website and follows each step of the script.

<Tabs stacked>
	<TabsBar>
		<TabsBarItem id="URL">Go to a URL</TabsBarItem>
		<TabsBarItem id="element">Interact with a page element</TabsBarItem>
		<TabsBarItem id="wait">Wait for a page element</TabsBarItem>
        <TabsBarItem id="login">Log in to a website</TabsBarItem>

	</TabsBar>

	<TabsPages>
		<TabsPageItem id="URL">
        The beginning of all scripts specifies the URL the monitor should check.

        ```js
        //Visit `http://telco.nrdemo.com/`
        $webDriver.get("http://telco.nrdemo.com/");  
        ```
        </TabsPageItem>
		<TabsPageItem id="element">
        Once your monitor goes to a URL, it scans for then interacts with the page's available elements.
            
        ```js
        $webDriver.get("http://telco.nrdemo.com/").then(function(){
        //Find a link with display text that matches `About`
            return $webDriver.findElement($selenium.By.linkText("About")).click();
        //Click the link that matches `About`
        }).then(function(){
            return $webDriver.findElement($selenium.By.partialLinkText("Home")).click();
        });
        ```
		</TabsPageItem>
		<TabsPageItem id="wait">
        Your script may need to wait an interval before reporting a failure, since images or dynamic elements can take time to generate on a page.

        ```js
        $webDriver.get("http://telco.nrdemo.com/").then(function(){
            return $webDriver.findElement($selenium.By.id("supportDropDown")).click();
        }).then(function(){
        //Wait up to 20000 seconds for the FAQ page to appear
            return $webDriver.waitForAndFindElement($selenium.By.id("supportFAQLink"), 20000).then(function(aboutPage){
                return aboutPage.click();
            })
        });
        ```
		</TabsPageItem>
        <TabsPageItem id="login">
        You can take your stored secure credentials to test certain page elements on your web apps. For scripting purposes, we recommend that you create new credentials unique to your script. Don't use your personal credentials or reuse credentials when running a monitor.

        ```js
        $webDriver.get("http://example.com/login.jsp").then(function(){
            return $webDriver.findElement($selenium.By.name("username")).sendKeys($secure.SECURE_USERNAME);
        }).then(function(){
        //Find the password field by specifying its name, then submits a secured password.
            return $webDriver.findElement($selenium.By.name("password")).sendKeys($secure.SECURE_PASSWORD);
        }).then(function(){
        //Find and click the login button.
            return $webDriver.findElement($selenium.By.xpath("//inp[@value='Login']")).click();
        });
        ```
		</TabsPageItem>
	</TabsPages>
